# Describe a Major P1 Production Incident - Kubernetes Liveness Probe Failure (STAR)

## Situation

While working as an SRE at Mastercard, we faced a P1 incident where a production deployment caused application availability issues.

After the deployment, some Kubernetes pods started continuously restarting because the liveness probe was failing, which caused the application instances to become unavailable.

The monitoring alerts showed increased transaction failures, and users were unable to complete their file transfer requests due to the application service disruption.

---

## Task

As the on-call SRE,
my responsibility was to identify the root cause, restore application availability quickly, coordinate with the application team, and minimize customer impact.

---

## Action

- I acknowledged the P1 alert and joined the incident bridge.
- I checked Dynatrace dashboards to understand the impact, error rate, and affected services.
- I verified the Kubernetes cluster health and checked pod status using `kubectl get pods`.
- I noticed that some pods were going into restart loops due to failed liveness probe checks.
- I checked pod events and logs using `kubectl describe pod` and `kubectl logs` to identify why the health check was failing.
- I found that after the deployment, the application startup time had increased, but the liveness probe timeout values were not adjusted accordingly.
- I coordinated with the application team and recommended increasing the probe timeout and rolling back the deployment if required.
- After applying the fix, I monitored pod health, transaction success rate, application latency, and logs to confirm recovery.

---

## Result

The pods stabilized, application availability was restored, and transaction processing returned to normal.

As a preventive action, we reviewed Kubernetes health check configurations and improved deployment validation
to ensure readiness and liveness probes were properly tested before production releases.
