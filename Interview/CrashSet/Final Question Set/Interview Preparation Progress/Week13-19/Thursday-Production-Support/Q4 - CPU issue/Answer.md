# Interview Answer – How do you troubleshoot High CPU Utilization?

> **When I observe high CPU utilization, I first determine the scope of the issue using monitoring tools like Prometheus, Grafana, Datadog, or CloudWatch. I check whether the spike is affecting a single server, multiple servers, or an entire application, and correlate it with metrics such as CPU utilization, request rate, response time, latency, and 5xx errors to understand the business impact. Based on this, my initial hypothesis would be a traffic spike, a runaway process, a recent deployment, an application bug, a Java garbage collection issue, or resource-intensive database operations.**
>
> **Once I've confirmed the affected host, I log into the server and use `top` or `htop` to verify CPU utilization, identify idle CPU percentage, and review the load average. A high load average compared to the available CPU cores indicates that processes are waiting for CPU time. I then identify the process consuming the most CPU using `ps -eo pid,comm,%cpu --sort=-%cpu`. If it's a multi-threaded application such as Java, I perform thread-level analysis using `top -H -p <PID>` and, if required, correlate the busy thread with a thread dump using `jstack` to determine what the application is doing.**
>
> **Next, I review the application logs using `journalctl -u <service>` or centralized logging tools like Splunk or Datadog Logs to look for exceptions, infinite loops, failed jobs, timeout errors, or repeated retries. I also verify overall system health using `uptime` for load average, `vmstat` for CPU and memory pressure, `iostat` for disk I/O bottlenecks, and `free -m` to check whether memory exhaustion or swapping is indirectly contributing to high CPU usage.**
>
> **After confirming the system health, I investigate recent changes such as deployments, configuration updates, traffic increases, scheduled batch jobs, or inefficient database queries that could explain the CPU spike. This helps determine whether the issue is caused by infrastructure, workload, or the application itself.**
>
> **Once the root cause is identified, I take the appropriate corrective action. Depending on the findings, I may scale the application horizontally, restart a stuck service, roll back a faulty deployment, optimize database queries, tune the application, or terminate a runaway process after following the appropriate production change process. Finally, I verify that CPU utilization, application latency, and error rates have returned to normal, monitor the system for stability, perform a Root Cause Analysis (RCA), and improve monitoring, alerting, or capacity planning to prevent the issue from recurring.**
