# Linux Interview Follow-up Questions (System Resource Utilization)

---

# What does `top` show?

The `top` command provides a real-time overview of the system's health. It displays:

- CPU utilization
- Memory utilization
- Swap usage
- Load average (1, 5, and 15 minutes)
- Running, sleeping, stopped, and zombie processes
- CPU and memory usage by each process

It is usually the first command I use to determine which system resource is under pressure.

---

# What do the CPU fields in `top` mean?

Example:

```text
%Cpu(s): 12.5 us, 5.3 sy, 0.0 ni, 78.0 id, 3.5 wa, 0.2 hi, 0.3 si, 0.0 st
```

## us (User CPU)

Percentage of CPU time spent executing user applications.

Example:
- Java application
- Python script
- Nginx
- MySQL

High **us** generally means application processes are consuming CPU.

---

## sy (System CPU)

Percentage of CPU time spent executing kernel (operating system) tasks.

Examples:

- Process scheduling
- Memory management
- File system operations
- Network stack

High **sy** may indicate heavy kernel activity or excessive system calls.

---

## ni (Nice CPU)

Shows CPU time consumed by processes running with a modified **nice value** (priority).

Linux allows changing process priority using:

```bash
nice
renice
```

Example:

A backup job is started with

```bash
nice -n 10 backup.sh
```

Since it has a lower priority, CPU usage appears under **ni** instead of **us**.

---

## id (Idle CPU)

Percentage of CPU that is completely idle.

Example:

```
id = 90%
```

means

- CPU is free most of the time.
- The server is not CPU-bound.

High idle CPU usually indicates CPU is **not** the bottleneck.

---

## wa (I/O Wait)

Percentage of CPU time spent waiting for disk or storage I/O operations to complete.

High I/O wait usually indicates:

- Slow disks
- Heavy disk reads/writes
- Storage bottlenecks
- Database-intensive workloads

If **wa** is consistently high, I investigate further using:

```bash
iostat -x
iotop
vmstat
```

---

## hi (Hardware Interrupts)

CPU time spent handling hardware interrupts.

Examples:

- Disk controller
- Network Interface Card (NIC)
- Keyboard
- USB devices

Normally this value is very low.

High **hi** may indicate:

- Faulty hardware
- High network traffic
- Storage controller issues

---

## si (Software Interrupts)

CPU time spent handling software interrupts.

These are generated by the Linux kernel rather than hardware.

Examples:

- Network packet processing
- Kernel scheduling
- SoftIRQ processing

High **si** may occur on:

- Busy web servers
- High network traffic
- DDoS attacks
- High packet processing workloads

---

## st (Steal Time)

Steal time is mainly seen on virtual machines.

It represents the percentage of CPU time taken away by the hypervisor because another virtual machine is using the physical CPU.

Example:

You're running on AWS EC2.

If another VM on the same physical host consumes excessive CPU, your VM has to wait.

That waiting time appears as **st**.

High steal time indicates:

- CPU contention on the virtualization host.

Usually seen on:

- VMware
- KVM
- AWS EC2
- Azure VM
- Google Cloud VM

---

# Does `top` show disk usage?

## Answer

No.

`top` does **not** show filesystem utilization or disk usage.

It only displays **I/O wait (`wa`)**, which represents the percentage of CPU time waiting for disk I/O operations to complete.

A high I/O wait suggests storage performance issues, but `top` cannot identify:

- Which disk is affected
- Which process is responsible

For disk analysis, I use:

```bash
df -h
du -sh *
iostat -x
iotop
```

# What is Load Average?

## Answer

Load average represents the average number of processes that are:

- Running on the CPU
- Waiting to get CPU time
- Waiting in an uninterruptible state (usually waiting for disk I/O)

Linux displays load average over:

- Last **1 minute**
- Last **5 minutes**
- Last **15 minutes**

Example:

```text
Load average: 0.50 0.80 1.20
```

If a server has **4 CPU cores**:

- Load ≈ 4 → CPUs are fully utilized.
- Load < 4 → CPU has available capacity.
- Load > 4 → More processes are waiting than CPUs available, indicating resource contention.

A consistently high load average may indicate:

- High CPU usage
- Heavy disk I/O
- Too many running processes
- Resource bottlenecks

# What do you do if CPU utilization is high?

## Answer

If CPU utilization is high, I first identify the processes consuming the most CPU.

Commands:

```bash
ps aux --sort=-%cpu

top
```

After identifying the process, I determine whether it is caused by:

- Expected application workload
- High incoming traffic
- Scheduled batch jobs
- Infinite loops
- Runaway processes
- Misconfigured applications

I then review:

- Application logs
- System logs
- Monitoring dashboards

Based on the findings, I may:

- Restart the service
- Optimize the application
- Increase CPU resources
- Scale horizontally if needed

# Why do you use `free -m`?

## Answer

`free -m` is used to check memory and swap utilization.

It displays:

- Total memory
- Used memory
- Free memory
- Shared memory
- Buffers/cache
- Available memory
- Swap usage

Example:

```bash
free -m
```

If swap usage keeps increasing while available memory decreases, it may indicate:

- Memory pressure
- Memory leak
- Insufficient RAM

I then identify memory-intensive processes using:

```bash
ps aux --sort=-%mem
```

and investigate the application.

---

# Why do you use `df -h`?

## Answer

`df -h` checks filesystem utilization.

It displays:

- Filesystem
- Total size
- Used space
- Available space
- Percentage used
- Mount point

Example:

```bash
df -h
```

If a filesystem reaches 90–100% utilization, applications may fail to write logs or data.

To identify what is consuming the space, I use:

```bash
du -sh *
```

---

# Why do you use `du -sh *`?

## Answer

`du -sh *` shows the size of each file and directory in the current location.

Example:

```bash
cd /var

du -sh *
```

This helps identify large directories such as:

- Logs
- Backups
- Temporary files
- Application data

It is commonly used after `df -h` indicates a filesystem is running out of space.

---

# Which commands do you use to investigate disk I/O performance?

## Answer

If I suspect a disk performance issue, I use the following commands.

## `iostat -x`

Provides:

- Disk utilization (%util)
- Read/write throughput
- Average wait time (await)
- Queue length
- Service time

Useful for identifying storage bottlenecks.

---

## `iotop`

Displays:

- Processes performing disk reads
- Processes performing disk writes

Useful for identifying which application is generating heavy disk I/O.

---

## `vmstat`

Provides:

- CPU statistics
- Memory statistics
- Paging activity
- Context switches
- I/O statistics

Useful for determining whether the bottleneck is related to CPU, memory, or storage.

# Troubleshooting Flow

```text
System is slow
        │
        ▼
Run top
        │
        ├── High CPU
        │       │
        │       ├── ps aux --sort=-%cpu
        │       └── Investigate application/logs
        │
        ├── High Memory
        │       │
        │       ├── free -m
        │       ├── ps aux --sort=-%mem
        │       └── Check for memory leak
        │
        ├── High I/O Wait
        │       │
        │       ├── iostat -x
        │       ├── iotop
        │       └── vmstat
        │
        └── Disk Full
                │
                ├── df -h
                └── du -sh *
```

This systematic approach demonstrates not only knowledge of Linux commands but also the reasoning behind choosing each command and how to proceed based on the results.
